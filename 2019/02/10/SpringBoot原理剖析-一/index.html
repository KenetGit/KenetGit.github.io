SpringBoot原理剖析(一)

发表于 2019-02-10

本篇介绍SpringBoot应用启动背后的原理。

  • SpringBoot的自动装配
  • SpringBoot的启动执行过程

SpringBoot为java web提供了更加便捷的开发，为常见开发所需要的配置提供了自动配置的功能，让开发者从繁琐的项目配置中解放出来。SpringBoot并不是新的开发框架，而是基于Spring的增强，是基于Spring的封装，理解SpringBoot的工作原理有助于后期开发中解决问题。

SpringBoot的自动装配

理解SpringBoot启动原理，从SpringBoot启动入口开始，因为SpringBoot内嵌了Tomcat和Jetty服务器，且做了自动化的部署，所以运行SpringBoot项目只需要执行一个简单的main入口方法。

虽然定义使用了多个Annotation进行了原信息标注，但实际上重要的只有三个Annotation：

  • @SpringBootConfiguration
  • @EnableAutoConfiguration
  • @ComponentScan

我们来看看这三个注解是怎么实现的，都实现了些什么。

  • @SpringBootConfiguration，可见@SpringBootConfiguration应用了@Configuration

而@Configuration的实现中则应用了@Component，说明这本身是一个Bean。

如何使用@Configuration和@Component，以下举个简单的例子。

@Component声明这是一个组件类，以此通知Spring要为这个类创建bean，@Bean注解告诉Spring这个方法将会返回一个对象，这个对象要注册为Spring应用上下文中的bean。通常方法体中包含了最终产生bean实例的逻辑。两者的目的是一样的，都是注册bean到Spring容器中。@Component通常是通过类路径扫描来自动侦测以及自动装配到Spring容器中，而@Bean注解通常是我们在标有该注解的方法中定义产生这个bean的逻辑。

可见，使用@Component时是在类声明处，即源码的类名之上；而使用@Bean则更加灵活，只需要在返回这个对象的方法上使用，通常使用第三方类的时候由于不知道源码或者不可能在源码上添加@Component，就只能使用@Bean来创建bean实例，注入到容器中。

而@Configuration搭配@Bean使用，作用就相当于基于xml文件配置bean；@Configuration就相当于Spring配置文件中的<beans>标签，里面可以配置bean，@Bean相当于Spring配置文件中的<bean/>标签可以在Spring容器中注入一个bean。

  • @ComponentScan

@ComponentScan的功能就是自动扫描并加载符合条件的组件（比如@Component和@Repository等）或者bean定义，最终将这些bean定义加载到IoC容器中。可以通过basePackages等属性来细粒度的定制@ComponentScan自动扫描的范围，如果不指定，则默认Spring框架实现会从声明@ComponentScan所在类的package进行扫描。

SpringBoot的启动类SpringBootStarterApplication放在root package，即@ComponentScan所在类的package，所以我们没有通过任何配置指定包的扫描范围，SpringBoot就自动完成扫描和装配bean。

  • @EnableAutoConfiguration

查看一下源码实现，其中出现了两个新的注解。@EnableAutoConfiguration的功能是借助@Import的帮助，将所有符合自动配置条件的bean定义加载到IoC容器。

  • @AutoConfigurationPackage，源码中引入了Registrar.class。

而Registrar.class的功能是注册了一个Bean的定义；

new PackageImport(metadata).getPackageName()，其实返回了当前主程序类的 同级以及子级的包组件。

这也就是为什么，我们要把SpringBoot的启动类SpringBootStarterApplication放在项目的root package下。

  • @Import({AutoConfigurationImportSelector.class})

该类实现了DeferredImportSelector，而后者则继承了ImportSelector。

ImportSelector的一个方法为：selectImports，可以看到第九行，它其实是去加载 public static final String FACTORIES_RESOURCE_LOCATION = “META-INF/spring.factories”外部文件，这个外部文件，有很多自动配置的类。(在类路径下查看)。

@EnableAutoConfiguration自动配置：从classpath中搜寻所有的META-INF/spring.factories配置文件，并将其中org.springframework.boot.autoconfigure.EnableutoConfiguration对应的配置项通过反射实例化为对应的标注了@Configuration的JavaConfig形式的IoC容器配置类，然后汇总为一个并加载到IoC容器。

SpringBoot的启动执行过程

实例初始化

如果我们使用的是SpringApplication的静态run方法，那么，这个方法里面首先要创建一个SpringApplication对象实例，然后调用这个创建好的SpringApplication的实例方法。在SpringApplication实例初始化的时候，它会提前做几件事情：

  • 根据classpath里面是否存在某个特征类（org.springframework.web.context.ConfigurableWebApplicationContext）来决定是否应该创建一个为Web应用使用的ApplicationContext类型
  • 使用SpringFactoriesLoader在classpath查找并加载所有可用的ApplicationContextInitializer。
  • 使用SpringFactoriesLoader在classpath中查找并加载所有可用的ApplicationListener
  • 推断并设置main方法的定义类

开始执行

SpringApplication实例初始化完成并且完成设置后，就开始执行run方法的逻辑。首先遍历执行所有通过SpringFactoriesLoader可以查找到并加载的SpringApplicationRunListener。调用它们的started()方法，告诉这些SpringApplicationRunListener

至此，SpringBoot应用就启动完毕了。
