Java并发编程：基础和原理

发表于 2019-02-07

内存模型

核心：针对内存可见性问题学习内存模型！！！

内存模型与线程间通信

共享变量：实例域，静态域和数组元素都存储在堆内存中，在线程间共享。

局部变量：方法定义参数和异常处理器参数不会在线程间共享，即线程独有的。

JMM（Java内存模型），控制Java线程之间的通信。

JMM定义线程好主内存之间的抽象关系：线程之间的共享变量存储在主内存中，每个线程都有一个私有的本地内存，本地内存中存储了该线程以读/写共享变量的副本。

本地内存（工作内存）是JMM的一个抽象概念，实际上并不存在，它涵盖了缓存、写缓冲区、寄存器以及其他的硬件和编译器优化。

线程A与线程B的通信步骤：

  1. 线程A把本地内存A中更新过的共享变量刷新到主内存中去
  2. 线程B到主内存中去读线程A之前已更新过的共享变量

JMM就是通过控制主内存与每个线程的本地内存之间的交互，来为Java程序提供内存可见性保证。

源代码与指令序列的重排序

在执行程序时，为了提高性能，编译器和处理器常常会对指令做重排序。重排序分3种类型。

1） 编译器优化的重排序：编译器在不改变单线程程序语义的前提下，可以重新安排语句的执行顺序。

2） 指令集并行的重排序：现代处理器采用了指令级并行技术来将多条指令重叠执行。如果不存在数据依赖性（对同一个变量的读写关系），处理器可以改变语句对应机器指令的执行顺序。

3） 内存系统的重排序：由于处理器使用缓存和读/写缓冲区，这使得加载和存储操作看上去可能是在乱序执行。

JMM的编译器重排序规则会禁止特定类型的编译器重排序，对于处理器重排序，JMM的吹起重排序规则会要去Java编译器在生成指令序列时，插入特定类型的内存屏障指令，通过内存屏障指令来禁止特定类型的处理器重排序。JMM正是这样通过禁止特定类型的编译器重排序和处理器重排序，为程序员提供一致的内存可见性保证。

synchronized

synchronized也被称为重量级锁，Java 1.6为了减少获得锁和释放锁带来的性能消耗而引入的偏向锁和轻量级锁，以及锁的存储结构和升级过程。使用synchronized实现同步的基础：每个对象都可以作为锁，具体表现为以下3种形式。

  • 对于普通同步方法，锁是当前实例对象
  • 对于静态同步方法，锁是当前类的Class对象
  • 对于同步方法块，锁是synchronized括号里配置的对象

1. synchronized在JVM中的实现原理：

JVM基于进入和退出monitor对象来实现方法同步和代码块同步，二者的实现细节还有差异：代码块的同步是使用monitorenter和monitorexit指令实现的，而方法不同则是另一种实现方式（使用这两个指令也是可以实现方法同步的）。

monitorenter指令是在编译后插入到同步代码块的开始位置，而monitorexit是插入到方法结束和异常处，JVM要保证每个monitorenter必须有对应的monitorexit与之配对。任何对象都有一个monitor与之关联，当一个monitor被持有之后，它将处于锁定状态。线程执行到monitorenter指令时，就会尝试获取对象所对应的monitor的所有权，即尝试获取对象锁。

synchronized用到的锁是存放在Java的对象头中的Mark Word字段，该对象头字段|还存放了hashcode和分代年龄等信息。

锁状态 25bit 4 bit 1bit 是否偏向锁 2bit 锁标志位
无锁状态 对象的hashCoe 对象分代年龄 0 01

Mark Word存储的数据结构和信息会随着锁标志位的变化而发生变化。

锁状态 存储内容 标志位
轻量级锁 指向栈中锁记录的指针 00

2. 锁的升级

Java 1.6引入了偏向锁和轻量级锁，并且锁有4种状态，级别从低到高依次是：无锁状态、偏向锁状态、轻量级锁状态和重量级锁状态，这几个状态会随着竞争情况逐渐升级。锁可以升级但不能降级。

  • 偏向锁

    目的是消除数据再无竞争情况下的同步原语。偏向锁会偏向第一个获取它的线程，如果在执行的过程中，该锁没有被其他线程获取，则当前获得锁的线程不会进行同步。

    当锁对象第一次被线程获取的时候，线程使用CAS操作把这个锁的线程ID记录到对象头的Mark Word中，同时置偏向标记位为1。以后线程在进入和退出同步块时不需要进行CAS操作来加锁和解锁，只需要简单地测试一下对象头的Mark Word里是否存储着指向当前线程的偏向锁（即线程ID），如果测试成功则表示线程已经获得了锁。

    如果线程使用CAS操作时失败则表示该锁对象上存在竞争并且这个时候另一个线程获得偏向锁的所有权。当到达全局安全点时，持有锁的线程被暂停，重新设置对象头的Mark Word为无锁状态（不是偏向锁的状态，从而接下来发生锁升级），然后恢复线程，继续执行。

  • 轻量级锁

    加锁：线程执行同步块之前，JVM会先在当前线程的栈帧中创建用于存储锁记录的空间，并将对象头中的Mark Word复制到锁记录中，官方称为Displaced Mark Word。然后线程尝试使用CAS将对象头中的Mark Word替换为指向锁记录的指针。（从无锁升级为轻量级锁）。如果成功，当前线程获得锁；否则表示存在锁竞争，当前线程使用自旋来等待获得锁。

    解锁：使用CAS操作将Displaced Mark Word替换回到对象头，如果成功则表示没有竞争发生。如果失败，表示锁存在竞争，锁就会膨胀成为重量级锁。

    因为自旋会消耗CPU，所以锁升级为重量级锁之后不会降级到轻量级锁，以免发生严重的自旋。重量级锁中，试图获得锁的线程会阻塞直到锁被释放后才会被唤醒。

  • 锁的优缺点

锁 优点 缺点 适用场合
偏向锁 加、解锁无额外消耗 如果存在竞争，
则会有额外的锁撤销消耗		 只有一个线程访问同步块
轻量级锁 竞争的线程不阻塞，
提高响应速度		 自旋消耗CPU资源 追求响应时间
重量级锁 不适用自旋，不会消耗CPU 线程阻塞，响应时间较慢 追求吞吐量

volatile

1. volatile 读-写的内存语义

理解volatile特性的一种方式，是把对volatile变量的单个读/写，看成是使用同一个锁对这些单个读/写操作做了同步。从内存语义来讲，volatile的写-读与锁的释放-获取具有相同的内存效果：volatile写和锁的释放具有相同的内存语义；volatile读与锁的获取具有相同的内存语义。

  • 当写一个volatile变量时，JMM会把该线程对应的本地内存中的共享变量值刷新到主内存。
  • 当读一个volatile变量时，JMM会把线程对应的本地内存中的值置为无效，然后重新从内存中读取共享变量。

线程之间这种读写的一致性，实际上就是线程之间的进行了通信。

2. volatile 内存语义的实现

为了实现volatile的内存语义，JMM会分别限制编译器重排序和处理器重排序，JMM针对编译器中排序指定了volatile重排序规则表。

（规则表）

为了实现volatile的内存语义，编译器在生成字节码的时候，会在指令序列中插入内存屏障来禁止特定类型的处理器重排序。编译器是采取了比较保守的插入内存屏障的策略。

各种屏障的作用：

屏障 功能描述
StoreStore 可以保证在volatile写之前，其前面的所有普通写操作都已经对任意处理器可见了，即所有的普通写被刷新到主内存中。
StoreLoad 避免volatile写与后面可能有的volatile读/写操作重排序。JMM选择在每个volatile写后面插入一个StoreLoad屏障
LoadLoad 用来禁止处理器把前面的volatile读与后面的普通读重排序
LoadStore 用来禁止处理器把前面的volatile读与后面的普通写重排序

JMM插入内存屏障的保守策略：

  • 在每个volatile写操作的前面插入一个StoreStore屏障
  • 在每个volatile写操作的后面插入一个StoreLoad屏障
  • 在每个volatile读操作的后面插入一个LoadLoad屏障
  • 在每个volatile读操作的后面插入一个LoadStore屏障

双重检查锁定与延迟初始化

在Java多线程中，有时候需要采用延迟初始化类和延迟创建对象的开销。双重检查锁定是常见的延迟初始化技术，但使用它容易忽视一些重要的细节从而导致错误使用（非线程安全），下面将介绍如何使用线程安全的延迟初始化技术。

1. 非线程安全

这个解决方案要求JDK5或以上版本。使用volatile关键字修饰引用声明，重排序在多线程中将被禁止，从而保证线程安全的延迟初始化。

5. 基于类初始化的解决方案

这种方案是利用JVM的类加载过程的机制来解决的，因为JVM在类的初始化阶段会执行类的初始化，而在类的初始化期间，JVM会获取一个锁，这个锁可以同步多个线程对同一个类的初始化。在同一个类加载器下，一个类型只会被初始化一次。当一个线程获得锁并开始初始化一个类的时候，其他试图获得锁的线程将被阻塞，JVM在类加载过程的这一特性保证了类的初始化的线程安全性。

类的初始化：调用类的构造器<cinit>对静态代码块或者静态成员进行初始化，这是由虚拟机隐式完成的。注意类的构造器区别于构造函数（前者是在类加载过程中的初始化用到的，后者是创建类对象实例的时候用到的）。

因为JVM把静态内部类当做top-level类处理的，因此会把它从类内部独立出来完成类的加载过程并初始化它内部的静态成员。
